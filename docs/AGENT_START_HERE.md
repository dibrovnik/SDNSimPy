# AGENT_START_HERE

## Зачем создан этот файл

Репозиторий `SDNSimPy` в текущем состоянии небольшой и преимущественно плоский по структуре. Этот файл нужен как стартовая записка для ИИ-агента, чтобы он не начинал работу вслепую и не переносил в проект абстрактную структуру без учёта реальных файлов.

## Снимок текущей структуры репозитория

На момент подготовки файлов обнаружены следующие основные элементы:

- `SDN_Simulator.py` — основной крупный модуль, вероятный GUI-entrypoint и точка оркестрации симуляции.
- `DiscreteEvents.py` — вероятная модель дискретных событий.
- `FlowInformation.py` — данные по потокам/пакетам.
- `simulationResultInformation.py` — результаты симуляции и/или экран результатов.
- `PacketImportWindow.py` — импорт пакетов.
- `InfoSwitchWindow.py`, `InfoLinkWindow.py` — GUI-окна информации.
- `customizetopology.py` — настройка топологии.
- `ProgramaGrafos.py` — работа с графами/топологией.
- `Utilities.py`, `Threads.py` — служебные модули.
- `Graphs/Topology.json` — пример/исходник топологии.
- `Packets/Paquetes.pcap` — пакетный ввод.
- `Arch/` — ресурсы интерфейса.

## Что это означает для адаптации

- Текущий проект не разложен по пакетам `core/`, `models/`, `policy/` и т.д.
- Новую архитектуру нужно вводить постепенно и аккуратно.
- Высока вероятность, что сначала понадобится вынести вычислительное ядро из GUI-ориентированных модулей.
- Любая новая каталогизация должна учитывать совместимость с существующим запуском.

## Рекомендуемый первый проход

1. Прочитать `SDN_Simulator.py`, `DiscreteEvents.py`, `FlowInformation.py`, `simulationResultInformation.py`.
2. Построить `docs/ADAPTATION_MAP.md`:
   - что остаётся без изменений;
   - что расширяется;
   - что оборачивается адаптером;
   - что переносится в новый слой;
   - какие старые SDN-сущности остаются, но не участвуют в эксперименте.
3. Выделить границу между:
   - ядром симуляции;
   - моделью сети;
   - импортом данных;
   - визуализацией/GUI;
   - результатами.
4. Только после этого вводить новые доменные сущности:
   - `MessageClass`
   - `SecurityProfile`
   - `SecureMessage`
   - `PolicyVersion`
   - `ReplayWindow`
   - `PolicyManager`
   - `CryptoEngine`
   - `GatewayScheduler`
   - `MetricsCollector`

## Практический совет по структуре

Так как текущий репозиторий плоский, безопаснее сначала добавить новый изолированный слой, например:

- `docs/`
- `configs/`
- `tests/`
- пакет с новым исследовательским кодом

и лишь затем постепенно переносить/адаптировать существующие части в более чистую архитектуру.
